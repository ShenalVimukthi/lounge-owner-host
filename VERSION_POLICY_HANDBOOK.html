SmartTransit Backend Version Policy Handbook

SmartTransit Backend Version Policy Handbook

Last updated: 2026-03-05 Audience: New and existing backend developers

Purpose

This handbook defines the approved versions for backend development and explains why each version choice matters. The goal is simple:

  • predictable builds,
  • fewer environment-related bugs,
  • safer dependency upgrades,
  • faster onboarding for new team members.

If your local setup differs from this handbook, align to this document first.

1) Mandatory Version Baseline

Area Approved Version Why this version is selected
Go language 1.24.5 Matches go.mod, CI expectations, and current dependency compatibility. Prevents module/toolchain mismatch issues.
Docker build image golang:1.24-alpine Keeps container build environment aligned with Go target version while staying lightweight.
Docker runtime image alpine:latest Small runtime image footprint and fast startup for backend service containers.
OpenAPI spec 3.0.3 Stable and widely supported by tooling used in testing/documentation workflows.
Service port 8080 Standardized local and container port across scripts and deployment config.
Choreo component schema 1.2 Matches current deployment descriptor format used by this project.
Postman collection schema v2.1.0 Compatible with current Postman import/export and collaboration workflows.

4) Known Out-of-Scope Versions (Backend Repo)

The following are not pinned in this backend repository and must be controlled in the frontend/mobile repository:

  • Flutter SDK version
  • Android Gradle Plugin version
  • Gradle wrapper version
  • compileSdkVersion, targetSdkVersion, minSdkVersion
  • Android Build Tools version
  • Kotlin version
  • Android build JDK version
  • iOS deployment target / Xcode version

Why this is out-of-scope: these versions belong to app build pipelines, not backend build/runtime.

5) Rules for New and Existing Developers

  1. Use Go 1.24.5 locally.
  2. Do not manually edit dependency versions outside normal go mod workflow.
  3. Run checks before pushing: make check.
  4. Keep Docker and Go versions aligned (Go toolchain in go.mod and Dockerfile must match).
  5. Update this handbook in the same PR whenever version policy changes.

6) Safe Upgrade Process

When upgrading any core version:

  1. Create a small PR focused only on the version bump.
  2. Run go mod tidy, tests, and basic endpoint smoke tests.
  3. Check authentication, database, and payment-related flows.
  4. Update this handbook and TECH_STACK_AND_DEPENDENCIES.md together.
  5. Merge only after at least one teammate validates locally.

7) Quick Onboarding Checklist

  • Install Go 1.24.5.
  • Pull latest code.
  • Run make install-deps.
  • Configure .env from .env.example.
  • Start backend: make run.
  • Verify health endpoint: http://localhost:8080/health.

If your environment does not match expected behavior, first re-check your versions against this handbook.
